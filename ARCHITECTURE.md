# Архитектура, компромиссы и решения

---

## Архитектура

### Общий подход

Проект построен как **modular monolith** с чистым разделением на слои по принципам Clean Architecture:

```
core/domain          — чистые доменные сущности, никаких зависимостей
core/application     — use cases + порты (интерфейсы), зависят только от domain
infrastructure       — реализации портов (TON API, localStorage, Web Crypto)
features             — UI-фичи, каждая в своей папке
shared               — переиспользуемые UI-компоненты, утилиты, i18n
app                  — точка входа, providers, роутер, shell
```

**Зависимости идут строго сверху вниз:** `features` -> `core/application` -> `core/domain`. Infrastructure подключается через Dependency Injection в `app/providers`.

### Почему так

- **Тестируемость** — use cases тестируются без UI, infrastructure заменяется моками через порты
- **Заменяемость** — TON Center можно заменить на другой провайдер, не трогая бизнес-логику
- **Читаемость** — каждый файл < 200 строк, каждая папка < 7 файлов

### Управление состоянием

| Что | Где |
### Управление состоянием
|---|---|
| Сессия кошелька (unlocked secret) | Zustand (только в памяти, не персистится) |
| Баланс, транзакции | TanStack Query (кэш + автообновление) |
| Зашифрованный кошелек | localStorage через `BrowserWalletStore` |
| Доверенные адреса | localStorage через `BrowserRecipientBookStore` |
| UI-состояние форм | useState / useReducer в компонентах |

### Роутинг

TanStack Router с lazy-loaded страницами. Структура:

```
/              — стартовая (создать / импорт / разблокировать)
/create        — создание кошелька
/import        — импорт по seed-фразе
/unlock        — разблокировка паролем
/app/wallet    — дашборд (баланс + транзакции)
/app/send      — отправка TON
/app/receive   — получение (адрес + QR)
/app/settings  — настройки
```

Авторизованные маршруты (`/app/*`) защищены проверкой сессии в `AppShell`.

## Выбранные компромиссы

### 1. Хранение секрета

**Решение:** Seed-фраза шифруется через Web Crypto API (`PBKDF2 600K итераций + AES-256-GCM`) и хранится в localStorage. Расшифрованный секрет живет только в памяти текущей вкладки.

**Компромисс:** JavaScript-строки иммутабельны — невозможно надежно зануллить seed в памяти. Для testnet-задания это приемлемо. В production нужен WebAssembly-модуль для работы с секретами.

**Защита:**
- Auto-lock через 15 минут бездействия
- Ручной Lock wallet
- Backoff после серии неверных попыток ввода пароля
- При потере секрета в памяти — автоматический редирект на unlock

### 2. Подтверждение транзакции

**Решение:** После broadcast отслеживаем рост `seqno` кошелька, затем ищем новую outgoing-транзакцию в истории.

**Компромисс:** Это polling-подход, не event-driven. Возможен timeout если сеть медленная. Но для browser-only без backend это единственный вариант без WebSocket-подписок.

### 3. Rate limiting TON Center API

**Решение:**
- Увеличенные интервалы опроса (30с баланс, 45с транзакции)
- staleTime 30с — не рефетчим свежие данные
- retryDelay 5с при ошибках
- Автоматический retry при 429 (Too Many Requests) с задержкой 3с

**Компромисс:** Данные обновляются не мгновенно. Для кошелька с testnet-нагрузкой это нормально.

### 4. Стилизация

**Решение:** Tailwind CSS 4 с кастомной темой. Все стили — utility-классы в JSX, без CSS-модулей и отдельных файлов.

**Компромисс:** Длинные className-строки. Но зато нет CSS-файлов для поддержки, нет проблем с неймингом, стили видны прямо в компоненте.

### 5. Один кошелек на браузер

**Решение:** Поддерживается только один кошелек одновременно. Новый создание/импорт перезаписывает предыдущий.

**Компромисс:** Упрощает UX и архитектуру. Multi-wallet — логичное следующее улучшение.

## Логика защиты от подмены адреса (anti-address-swap)

Это ключевая security-фича приложения. При нажатии "Review transfer" выполняются проверки:

### Блокирующие ошибки (отправка невозможна)

| Проверка | Зачем |
|---|---|
| Невалидный адрес | Защита от опечаток и мусорного ввода |
| Адрес без testnet-маркера | Предотвращение отправки на mainnet-адрес |
| Некорректная сумма | Пустая, отрицательная, нечисловая |
| Недостаточный баланс | С учетом обязательного резерва 0.05 TON |

### Предупреждения (отправка возможна, но с предупреждением)

| Проверка | Зачем |
|---|---|
| `new-address` | Адрес не встречался ранее — возможна ошибка копирования |
| `raw-address` | Raw-формат без контрольной суммы — выше риск ошибки |
| `self-send` | Отправка самому себе — скорее всего ошибка |
| `similar-address` | Адрес похож на недавний, но отличается — возможна подмена |
| `uninitialized-recipient` | Контракт получателя не развернут — адрес мог быть сгенерирован, но никогда не использован |

### Нормализация

Перед проверками адрес нормализуется в canonical `testnet user-friendly` формат. Это значит:
- Raw-адрес конвертируется в user-friendly
- Проставляется testnet-флаг
- Bounceable/non-bounceable обрабатывается корректно

Пользователь всегда видит нормализованный адрес на этапе подтверждения.

### Доверенные адреса

Пользователь может сохранить адрес как "доверенный" в настройках или при отправке. Доверенные адреса не генерируют warning `new-address`. Это снижает alert fatigue при повторных переводах.

## Дальнейшие улучшения

### Безопасность
- WebAssembly-модуль для работы с приватными ключами (зануление памяти)
- Biometric unlock (WebAuthn)
- Hardware wallet support (Ledger)
- CSP hardening для production

### Функциональность
- Multi-wallet поддержка
- NFT и Jetton (токены) отображение
- Mainnet режим с переключателем
- Address book с именами
- Push-уведомления о входящих транзакциях
- Экспорт истории транзакций

### UX
- Optimistic UI для баланса после отправки
- Skeleton-загрузки вместо "Loading..."
- Swipe-жесты для мобильной навигации
- Тёмная/светлая тема

### Инфраструктура
- Playwright e2e тесты
- CI/CD pipeline (GitHub Actions)
- Sentry для мониторинга ошибок
- WebSocket-подписки вместо polling (через TON Center WS API)
- PWA (Service Worker + offline mode)

### Производительность
- Виртуализация списка транзакций (для 1000+ записей)
- Кэширование QR-кода
- Preload критических маршрутов
