# Architecture Decisions

## Цель

Зафиксировать технический стек, архитектурный подход и организационные ограничения по коду до начала реализации.

## Технологический стек

### Основа приложения

- `React` как UI-слой.
- `TypeScript` в `strict` mode как основной язык.
- `Vite` как сборщик и dev server.
- `pnpm` как пакетный менеджер.

Почему:

- стек быстрый в разработке;
- хорошо подходит для browser-only приложения;
- дает строгую типизацию и хороший DX.

### Навигация и данные

- `TanStack Router` для маршрутов и typed navigation.
- `TanStack Query` для blockchain-запросов, кэша и polling.
- `Zod` для форм, схем данных, env и storage payloads.
- `Zustand` только для малого client-side state.

Почему:

- роутинг и данные остаются типобезопасными;
- async state не смешивается с UI state;
- валидация и нормализация данных живет в одном понятном месте.

### TON и безопасность

- `@ton/ton` для работы с кошельком и сетью.
- `@ton/crypto` для mnemonic и crypto-операций.
- `@ton/core` подключаем только если потребуется для более низкоуровневых сущностей.
- `Web Crypto API` для локального шифрования seed-фразы.
- `TON Center testnet` как внешний blockchain provider.

Почему:

- это естественный стек для browser-only `TON testnet` wallet;
- ключевая криптография и подпись остаются на клиенте;
- не нужен собственный backend.

### Стили и UI

- `CSS Modules` для локальной изоляции стилей.
- `CSS variables` для токенов темы.
- собственный набор UI primitives вместо тяжелого UI kit.

Почему:

- это дает полный контроль над визуальной системой;
- легче соблюдать наши дизайн-ограничения;
- меньше зависимости от чужой дизайн-системы.

### Тестирование и качество

- `Vitest` для unit и integration тестов.
- `React Testing Library` для тестов UI-поведения.
- `MSW` для сетевых моков.
- `Playwright` для e2e-сценариев.
- `ESLint` + `Prettier` для качества и единообразия кода.

Почему:

- покрываем domain, UI и пользовательские сценарии;
- можно отдельно тестировать риск-логику и сетевые состояния;
- удобно поддерживать стабильное качество проекта.

## Архитектурный подход

Подход:

- один frontend-репозиторий;
- `modular monolith`;
- строгие границы между слоями;
- dependency direction направлен внутрь: от UI и infrastructure к domain/application.
- route-based архитектура с app shell для основной части приложения.

Основные слои:

- `domain`: сущности, value objects, бизнес-правила.
- `application`: use cases и порты.
- `infrastructure`: адаптеры к TON, storage, crypto, clipboard.
- `features`: пользовательские сценарии и composable UI-модули.
- `shared`: повторно используемые утилиты, UI primitives, config.

## Принципы проектирования

Обязательные инженерные принципы:

- следуем `SOLID`;
- один модуль отвечает за одну понятную ответственность;
- бизнес-логика не зависит от React;
- компоненты не должны знать детали blockchain API;
- storage и network реализуются как адаптеры, а не вызываются из UI напрямую;
- сложная логика выносится из компонентов в domain/application слой;
- типы и схемы не дублируются бесконтрольно в разных местах.

## Жесткие ограничения по структуре кода

Согласованные ограничения:

- не больше `200` строк в одном файле;
- не больше `7` файлов в одной папке;
- если файлов становится больше `7`, они группируются по подпапкам;
- если файл растет и начинает совмещать несколько ответственностей, он разбивается до начала следующего этапа разработки.

Практический смысл:

- не допускаем god-components;
- не допускаем перегруженных feature-папок;
- поддерживаем читаемость проекта с самого старта.

## Навигационная модель

Согласованная схема:

- публичная часть:
  - `/`
  - `/create`
  - `/import`
  - `/unlock`
- основная часть приложения внутри общего shell:
  - `/app/wallet`
  - `/app/send`
  - `/app/receive`
  - `/app/settings`

Почему:

- внешний уровень остается route-based и расширяемым;
- внутренняя часть ощущается как компактное wallet-приложение;
- удобно проектировать mobile-first shell с постоянной навигацией.

## Правила использования библиотек

- `TanStack Query` используем только для server state и polling.
- `Zustand` не используем как хранилище всей бизнес-логики.
- `Zod` используем как единый источник правил валидации, где это возможно.
- TON SDK оборачиваем в инфраструктурные сервисы, а не тащим напрямую в UI.
- формы и предупреждения проектируем так, чтобы risk-логика могла тестироваться вне React-компонентов.

## Что это дает проекту

- проект остается масштабируемым;
- проще расширять wallet-функциональность;
- проще добавлять новые проверки риска;
- проще поддерживать тесты;
- проще контролировать качество кода уже на уровне структуры.
