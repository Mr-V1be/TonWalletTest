# Foundation Decisions

## Scope, который не обсуждаем

Это уже задано ТЗ:

- browser-only web app без собственного backend;
- сеть `TON testnet`;
- self-custodial модель;
- создание и импорт кошелька;
- адрес, баланс, история, поиск;
- `Receive` с адресом и копированием;
- `Send` с адресом, суммой и отправкой;
- заметные предупреждения в рискованных сценариях;
- README с архитектурой, компромиссами и запуском.

## Инженерные принципы и ограничения

Зафиксировано:

- следуем `SOLID`;
- строим модульный и расширяемый frontend;
- UI, бизнес-логика и инфраструктура должны быть разнесены;
- в одном файле не больше `200` строк;
- в одной папке не больше `7` файлов;
- если файлов больше `7`, группируем их в подпапки по ответственности.

Практический смысл:

- не допускаем god-files и god-folders;
- снижаем связность;
- упрощаем поддержку и рост проекта.

## Технологический стек

Зафиксировано:

- `React + TypeScript + Vite`;
- `TanStack Router` для навигации;
- `TanStack Query` для server state и polling;
- `Zod` для схем и валидации;
- `Zustand` только для небольшого client/session state;
- `@ton/ton`, `@ton/crypto`, при необходимости `@ton/core`;
- `Web Crypto API` для шифрования;
- `CSS Modules` + `CSS variables` + собственные UI primitives;
- `Vitest`, `React Testing Library`, `MSW`, `Playwright`;
- `pnpm` как package manager.

## Архитектурный подход

Зафиксировано:

- приложение строим как `modular monolith`;
- внутренние границы: `domain`, `application`, `infrastructure`, `features`, `shared`;
- бизнес-логика не зависит от React и browser APIs;
- внешние интеграции подключаются через адаптеры и интерфейсы.

## Seed storage

Зафиксировано:

- seed-фраза хранится локально только в зашифрованном виде;
- шифрование делаем через `Web Crypto`;
- доступ к seed дается через passcode пользователя.

Почему:

- это адекватный компромисс между UX и безопасностью для тестового задания;
- не нужен backend;
- пользователь не теряет удобство после reload за счет возможности unlock по passcode.

## TON wallet и сеть

Зафиксировано:

- базовый wallet contract: `Wallet V5R1`;
- публичный provider: `TON Center testnet`;
- чтение баланса, состояния аккаунта и истории идет через публичный API;
- транзакция подписывается локально в браузере;
- в сеть отправляется уже подписанный payload;
- подтверждение получаем через polling.

Почему:

- это прямолинейная и жизнеспособная browser-only схема;
- она соответствует формату тестового задания и не требует backend.
